## 1. 合作遊戲的核心設計原理

### 1.1 為什麼「強制合作」難以設計？

研究發現，大多數「合作遊戲」其實並非真正的強制合作——熟練玩家通常可以一個人攻略，或少數玩家「帶」其他人。這被稱為 **Quarterback Problem（四分衛問題）**：最強的玩家壟斷決策，其他人成為執行者而非合作者。

**真正的強制合作需要達到以下條件之一或多個：**

| 條件 | 定義 | 成功範例 |
|------|------|---------|
| 資訊不對稱 | 每個玩家只能看到部分資訊 | Keep Talking and Nobody Explodes |
| 空間隔離 | 玩家在不同位置，無法到達對方的工具 | We Were Here, Overcooked |
| 行動經濟 | 同時需要執行的任務數量超過單人能力 | Lovers in a Dangerous Spacetime |
| 同步執行 | 需要多人同時輸入才能觸發事件 | Portal 2 Co-op, It Takes Two |
| 能力互補 | A 的能力恰好填補 B 的弱點 | Deep Rock Galactic |

### 1.2「原始人狩獵長毛象」的設計意涵

您提到的這個比喻非常精準，它指向一種特定的合作感：
- **規模不對等**：獵物/目標比任何單一玩家強大得多
- **角色有機分工**：不同玩家自然承擔不同責任（圍堵、吸引、攻擊、支援）
- **風險共擔**：失敗是集體的，不是某一人的錯
- **節奏：準備→執行→收穫**：有前期規劃、有執行危機、有完成感
- **非語言協作**：很多時候是靠本能反應配合，而非語言指揮

這種感覺最接近的現代遊戲類比：**Monster Hunter（怪物獵人）系列**。

## 2. 六大合作模式詳解

### 模式 A：非對稱角色合作（Class-Based Asymmetry）

**核心概念**：每個玩家扮演一個有獨特能力集合的角色，角色之間相互補足。

**強制合作的實現方式**：
- 玩家 A 的工具能解決玩家 B 無法解決的問題
- 玩家 B 的能力讓玩家 A 的工具得以發揮最大效果
- 如果 A 不在，B 就卡關；反之亦然

**設計細節：**
```
四個角色的理想互補關係（以狩獵主題為例）：

[追蹤者] ←→ [陷阱師]
看得見獵物的位置，但不能直接攻擊   能設置致命陷阱，但看不見獵物

[衝鋒者] ←→ [支援者]
吸引獵物注意力，但容易受傷          能治療隊友，但無法獨自吸引注意力
```

**優點：**
- 重玩性高（可以嘗試不同角色）
- 每個玩家都有「存在意義」
- 支援新手（技能門檻較低的角色讓初學者有貢獻感）

**缺點：**
- 美術成本高（4 個角色 × 各自的動畫集）
- 平衡設計複雜（一個角色 OP 就破壞整個生態）
- 需要玩家學習期

**Godot 4 技術難度：中偏高**  
原因：4 個不同能力集合需要模組化的技能系統設計

---

### 模式 B：任務流程合作（Process-Based / Workflow Asymmetry）

**核心概念**：完成目標需要一個「流程線」，每個玩家負責流程的不同環節，缺一不可。

**典型例子（Overcooked 模式）**：
```
目標：獵殺並處理長毛象

[誘導者]         [圍捕者]         [攻擊者]         [收割者]
把獵物引到陷阱 → 防止獵物逃脫 → 削減獵物體力 → 最終擊殺並收集資源
```

**強制合作的實現方式**：
- 任何一個環節失敗，整個流程就中斷
- 環節之間有物理距離或時間壓力
- 玩家無法離開自己的位置去幫另一個玩家

**優點：**
- 強制合作感最強烈（最接近長毛象比喻）
- 設計相對直觀
- 可以通過調整環節難度控制整體難度

**缺點：**
- 容易因一人失敗造成「掉隊感」（需要有失敗緩衝設計）
- 重複性較高
- 如果流程設計不好，會覺得「按照程序走」而非真正合作

**Godot 4 技術難度：中等**  
原因：主要是遊戲狀態管理，無需過於複雜的系統

---

### 模式 C：資訊不對稱合作（Information Asymmetry）

**核心概念**：每個玩家只能看到部分世界資訊，必須透過溝通拼湊完整圖像。

**實現方式（針對 4 人）：**
```
玩家 A：只能看到地圖上半部，能看到出口位置
玩家 B：只能看到地圖下半部，能看到危險的位置
玩家 C：能看到所有怪物的血量，但看不到地圖
玩家 D：能看到所有玩家的體力，但看不到怪物
```

**優點：**
- 幾乎不需要複雜的美術資源（資訊差靠 UI 表達）
- 強迫語言/非語言溝通
- 獨特體驗，市場上相對少見
- 適合線上遊戲（因為玩家本來就在不同螢幕）

**缺點：**
- 如果玩家不溝通，遊戲完全無法推進（可能令不喜歡溝通的玩家挫折）
- 需要精心設計資訊的「粒度」（太少資訊令人崩潰，太多資訊沒有合作感）
- UI 設計複雜度高

**Godot 4 技術難度：中等（UI 複雜，核心邏輯不難）**

---

### 模式 D：空間依賴合作（Spatial Interdependence）

**核心概念**：玩家被物理上限制在不同空間，需要對方在特定位置做特定事，才能讓自己的任務推進。

**範例：**
```
玩家 A 在橋的這端按住機關 → 橋才會展開 → 玩家 B 才能過橋去啟動另一個機關
同時，玩家 C 在外面防止敵人打斷 A，玩家 D 在遠處偵察下一個目標
```

**優點：**
- 直觀易理解（「你在那邊幫我按住，我才能過去」）
- 場景設計驅動合作（合作感來自關卡本身，不需要複雜規則）
- 視覺反饋強（看得到對方在做什麼）

**缺點：**
- 關卡設計工作量巨大
- 每個關卡都需要新的合作謎題設計
- 較難做到「有機合作」，容易變成「解謎流程」

**Godot 4 技術難度：中等（場景機關邏輯）**

---

### 模式 E：共享資源合作（Shared Economy）

**核心概念**：玩家共享有限資源（血量、能量、物資），個人的揮霍直接損害集體。

**實現方式：**
```
共享體力槽：任何玩家受傷，所有人都承受部分傷害
共享行動點數：每回合全隊共 8 個行動點，玩家要商量分配
共享資源：物資需要合理分配，囤積會讓其他人沒有工具
```

**優點：**
- 強迫玩家協商，產生社交張力（好的那種）
- 不需要複雜的角色設計
- 設計失敗懲罰有意義（「你亂跑我們都掛了！」）

**缺點：**
- 容易因一個玩家失誤而產生「怪罪文化」
- 需要精心校調資源量，否則要麼太鬆（無所謂）要麼太緊（挫折）

**Godot 4 技術難度：低至中等（狀態同步問題）**

---

### 模式 F：動態角色輪換合作（Dynamic Role Rotation）

**核心概念**：玩家的角色/能力在遊戲過程中改變，需要即時適應和協調。

**範例：**
```
輪到玩家 A 拿到「長矛」→ 其他人要掩護他靠近獵物
之後輪到玩家 B 拿到「火把」→ 其他人要改變陣型配合
```

**優點：**
- 所有玩家都嘗試所有角色，沒有「坐冷板凳」問題
- 高重玩性
- 減少平衡壓力（因為每個人都會輪到各種角色）

**缺點：**
- 學習曲線較陡（要掌握多種角色）
- 合作邏輯更複雜
- 需要設計清晰的「角色切換指示」

**Godot 4 技術難度：中高（動態狀態管理）**

## 3. 對標遊戲深度分析

### 3.1 Deep Rock Galactic（2020，Ghost Ship Games）

**類型**：模式 A（非對稱角色合作）  
**玩家數**：1-4 人  
**核心合作機制**：

| 職業 | 獨特工具 | 其他人需要他的原因 |
|------|---------|-----------------|
| 鑽探者 (Driller) | 能在任何地方挖隧道 | 當隊伍被岩壁困住 |
| 工程師 (Engineer) | 能建造平台和砲台 | 讓其他人到達高處 |
| 偵察兵 (Scout) | 能用鉤爪到達任何地方，帶照明 | 黑暗地圖的眼睛 |
| 機槍手 (Gunner) | 最強火力，能建造繩索 | 火力壓制，運送隊友 |

**設計亮點（值得學習）：**
1. **自然分工**：不強制玩家「你必須做這個」，而是讓地形和情況自然產生分工需求
2. **死亡設計**：倒地後需要隊友救援（創造保護互依）
3. **「Rock and Stone」文化**：遊戲內建喊話系統，強化社群感

**美術規模（參考）：**
- 4 個高度差異化角色（各有獨特外形）
- 敵人：約 30+ 種
- 環境：4 個大型生物群落
- 預算估計：数百萬美元（AAA 規模，不可直接參考）

---

### 3.2 Overcooked 2（2018，Team17）

**類型**：模式 B（任務流程合作）  
**玩家數**：1-4 人  
**核心合作機制**：

空間分隔 + 任務鏈 + 時間壓力  
每個廚師只能在特定區域操作，食物必須在正確的工序順序下製作。

**設計亮點（值得學習）：**
1. **關卡即設計**：不同地圖有移動的工作台、分裂的地圖等，關卡本身製造合作難題
2. **即時反饋**：訂單進度條、完成音效讓每次成功都有即時滿足感
3. **清楚的失敗原因**：玩家很清楚「剛才誰犯了什麼錯」，便於改進

**對本專案的啟示**：  
如果做「狩獵」主題，可以把「獵物處理流程」設計成 Overcooked 式的鏈式任務。

---

### 3.3 It Takes Two（2021，Hazelight Studios）

**類型**：模式 C+D（資訊 + 空間依賴）  
**玩家數**：固定 2 人  
**核心設計原理**：每個章節引入全新的力學，玩家 A 和玩家 B 得到互補的能力，只能共同完成謎題。

**對 4 人擴展的設計建議（業界分析）：**
```
2 人版本：A[錘子] + B[釘子]
4 人版本：可以分「功能對」：
  對 1：A[錘子] + B[釘子] 負責結構破壞
  對 2：C[繩索] + D[拉力] 負責物件移動
  最終謎題需要兩對同時操作
```

---

### 3.4 Lovers in a Dangerous Spacetime（2015，Asteroid Base）⭐ 本專案主要參考方向

**類型**：模式 E（共享資源）+ 空間依賴  
**玩家數**：1-4 人  
**核心機制**：一艘飛船有多個戰鬥站（引擎、護盾、武器、超級武器），玩家需要不斷在不同站點之間跑動，沒有人能獨自控制全部。

#### 為什麼它有效——業界分析（Game Developer 研究摘要）

| 設計支柱 | 機制 | 對合作的影響 |
|---------|------|------------|
| **站點瓶頸** | 可操控站點數量 > 玩家數量 | 永遠有人手不夠，強制即時溝通 |
| **無固定角色** | 所有玩家都是通才，可操作任何站點 | 防止「坐冷板凳」，所有人持續參與 |
| **共享物理空間** | 單一畫面，所有人在同一環境可見 | 合作行為「看得見」，溝通更直覺 |
| **自發式合作** | 不規定怎麼分工，由玩家自行發展策略 | 每次遊玩都有不同的合作模式 |
| **成功靠溝通** | 快反應不是決定因素，協調才是 | 降低技術門檻，新手也能貢獻 |

**對本專案的直接參考價值很高**，因為：
- 「4 人在同一個環境內分工」的結構感
- 沒有固定角色，玩家需要即時判斷「現在誰去哪裡」
- 失敗是集體的，沒有個人的「不服輸」
- **關鍵洞察**：遊戲的成功公式是「*系統數量 > 玩家人數*」，這個不等式製造了永遠不夠用的壓力感

---

### 3.5 Keep Talking and Nobody Explodes（2015，Steel Crate Games）

**類型**：模式 C（資訊不對稱）  
**玩家數**：1（拆彈者）+ 多人（手冊持有者）

**設計啟示**：  
這個遊戲證明了「資訊分割」本身就是強力的合作機制。  
應用到本專案：可以讓不同玩家看到不同的「獵物狀態指示器」，只有集合所有資訊才能做出最佳決策。

---

### 3.6 Monster Hunter Series（Capcom）

**類型**：模式 A + B 混合  
**最接近「長毛象」氛圍的遊戲**

**核心體驗分析：**
1. **準備階段**（規劃）：選擇裝備、配合隊伍構成、制定策略
2. **執行階段**（狩獵）：實際戰鬥中的即時合作
3. **成果階段**（收割）：打倒後的資源收集

**為什麼這個氛圍如此特別：**
- 獵物有真實的行為模式和弱點
- 合作不是規定好的，而是自然產生的
- 每次狩獵即使相同的獵物也有不同的結果

**Godot 4 可行性評估**：完整的 MH 規模是不可能的，但核心的「獵物行為 + 玩家協作」系統是可行的。

## 4. itch.io 市場調查

### 4.1 市場現狀

根據 itch.io 搜尋結果（標籤：4-player + local-co-op + top-rated）：

**發現的相關遊戲：**

| 遊戲名稱 | 類型 | 合作機制 | 特點 |
|---------|------|---------|------|
| Launch Break | 太空派對遊戲 | 角色分工 | 飛船即將墜毀，4 人分工解決危機 |
| Lovers in a Dangerous Spacetime | 太空射擊 | 共享資源 | 獨立遊戲的合作經典 |
| Chessplosion | 戰術炸彈 | 競合 | 4 人戰術博弈 |
| Amphibia: Calamity Beatdown | 動作打鬥 | 並肩作戰 | Beat 'em up 風格 |

### 4.2 市場空缺分析

**4 人強制合作（非競爭）在 itch.io 上相對稀少**，尤其是：
- 「狩獵/捕獵」主題的合作遊戲（幾乎是空白）
- 強調「所有人都必須參與」而非「強者帶弱者」的設計
- 兼具動作要素和策略協調的遊戲

**這意味著**：如果本專案做得好，市場差異化明顯。

### 4.3 itch.io 上成功的小型合作遊戲共同特點

1. **清楚的視覺識別**：玩家角色顏色明確，不會搞混
2. **快速上手**：教學不超過 2 分鐘（本地多人遊戲的通常條件）
3. **失敗不懲罰太重**：重新嘗試的成本低
4. **小而精**：1-3 小時的核心體驗，而非龐大的內容量

## 5. 美術資源需求分析（各方向）

### 前提：本遊戲為 2D，Godot 4 引擎

---

### 方向一：像素風格（Pixel Art）

**適合的合作模式**：A（角色類）、B（流程類）、E（資源類）

**核心資源清單（4 人角色合作遊戲）：**

#### 角色類資源
```
主角色（×4 個不同角色）
每個角色需要以下動畫（幀數視難度而定）：
- 待機 (Idle)：6-8 幀
- 行走 (Walk)：8-12 幀
- 跑步 (Run)：6-10 幀
- 攻擊 1（普通攻擊）：6-8 幀
- 攻擊 2（特殊技能）：8-12 幀
- 受傷 (Hurt)：4-6 幀
- 倒地/死亡 (Dead/Down)：6-8 幀
- 勝利 (Victory)：4-6 幀
- 起身 (Revive)：6-8 幀

小計：每個角色約 54-78 幀
4 個角色合計：約 216-312 幀的像素畫
```

#### 敵人/獵物類資源（最小可行版本）
```
主要獵物（類長毛象）：
- 待機：8-12 幀
- 行走/奔跑：8-12 幀
- 攻擊模式 1-3：各 6-10 幀
- 受傷反應：4-6 幀
- 倒地/死亡：8-12 幀
- 衝撞/特殊行為：8-12 幀

小計：主要獵物約 60-80 幀

次要敵人（如果有）×2-3 種：
每種約 30-40 幀
```

#### 環境資源
```
地形 Tileset（1 個環境主題，最小可行版）：
- 地板磚片：8-16 種變化
- 牆壁磚片：8-16 種變化
- 背景裝飾：20-30 個物件
- 前景裝飾：10-15 個物件
- 互動物件（陷阱、機關等）：每種約 4-8 幀

背景層（視差捲動）：
- 遠景：1-2 層
- 中景：1-2 層
- 近景：1 層
```

#### UI 資源
```
HUD 元素：
- 4 個玩家血量/狀態條
- 技能冷卻圖示（每個角色 2-3 個技能）
- 共享資源指示器（如有）
- 獵物血量/狀態條
- 任務提示框
- 復活指示器

選單相關：
- 主選單背景
- 角色選擇畫面（4 個角色展示）
- 結算畫面
- 各種按鈕圖示
```

#### VFX（視覺特效）
```
戰鬥特效：
- 攻擊命中粒子
- 受傷閃光
- 能力施放特效（每個角色 1-2 種）
- 死亡/倒地特效

環境特效：
- 塵土/踩踏粒子
- 陷阱觸發特效
- 勝利特效

UI 特效：
- 血量降低動畫
- 技能就緒提示
```

**總估算（像素藝術，16x16 ~ 32x32 像素）：**

| 類別 | 估計幀/物件數 | 難度 |
|------|------------|------|
| 4 角色動畫 | 250-320 幀 | 高 |
| 主要獵物動畫 | 60-80 幀 | 中高 |
| 次要敵人（×3） | 90-120 幀 | 中 |
| 環境/地圖 | 60-90 物件 | 中 |
| UI | 40-60 元素 | 低 |
| VFX | 20-30 套 | 中 |
| **總計** | **約 520-700 個獨立資源** | — |

**時間估算（有繪畫基礎的獨立開發者）：**
- 每幀像素畫：約 0.5-2 小時（依複雜度）
- 估計總時間：**400-800 小時**（約 3-6 個月全職）

### 方向二：手繪/剪影風格（Hand-Drawn / Silhouette）

**適合的合作模式**：A（角色類）、D（空間依賴）

**最佳參考案例**：《空洞騎士》（Hollow Knight）的黑色剪影風格

**核心特點（剪影風格的優勢）：**
- 角色可以是純黑剪影 + 少量顏色細節，**大幅降低繪製複雜度**
- 動畫更容易因為細節少而流暢
- 視覺對比度高，玩家容易識別自己的角色
- 美術門檻比全彩像素畫更低，但視覺衝擊力更強

**資源清單差異（vs 像素風格）：**

| 類別 | 像素風格 | 剪影風格 | 差距 |
|------|---------|---------|------|
| 角色繪製工作量 | 100% | 60-70% | 省 30-40% |
| 動畫複雜度 | 中高 | 中等 | 降低 |
| 環境繪製 | 中等 | 低（黑暗背景） | 省 40-50% |
| 識別性 | 依靠顏色 | 依靠剪影形狀 | 需要更獨特的外形設計 |
| 整體美術成本 | 100% | 約 60-70% | 節省 30-40% |

**剪影風格特別注意：**
- 4 個角色的剪影外形必須**高度差異化**（不能讓玩家搞混）
- 建議：一高一矮、一瘦一壯，加上不同的標誌性武器外形
- 顏色光暈/後光（Glow）可以成為角色識別的輔助手段

---

### 方向三：簡約幾何/低多邊形風格

**適合的合作模式**：C（資訊不對稱）、F（動態輪換）

**核心概念**：角色用簡單幾何形狀（圓形、三角形、方形）表示，用顏色區分。

**優點：**
- 美術成本極低（可以是一人短期完成）
- 允許更多精力放在機制設計上
- 抽象風格在 itch.io 有一定市場（如 Among Us）

**缺點：**
- 沒有視覺衝擊力，難以在第一眼吸引玩家
- 商業化潛力較低
- 需要非常強的機制來補償美術的不足

**資源清單（最小版本）：**
```
角色（×4）：
- 只需要基本形狀 + 顏色變化
- 動畫：縮放/旋轉動畫（不需要逐幀）

環境：
- 幾何形狀組成的障礙物
- 程序生成或簡單手動設計

UI：
- 清晰的數字/條狀 UI（設計重點）
```

**估計製作時間：50-150 小時（含設計）**

### 6.2 實際製作成本（獨立開發視角）

**假設：一名開發者，有中等美術能力**

#### 像素風格最小可行版本（MVP）

總計：約 800 小時（約 5 個月）
```

### 6.3 隱藏成本（常被忽視）

| 隱藏成本 | 說明 | 估算 |
|---------|------|------|
| 音效/音樂 | 免費音源 vs 委外製作 | $0-$500 |
| 工具授權 | Godot 免費，Aseprite $20，Spine $99+ | $20-$200 |
| 遊戲測試 | 需要 4 個玩家，約定時間 | 時間成本 |
| Bug 修復超時 | 通常增加 20-30% 的開發時間 | +160-240h |
| 多人網路架構（若需要）| 線上多人難度倍增 | +200-400h |

**⚠️ 重要：本地 vs 線上多人**  
這是最關鍵的技術決策之一：
- **本地多人（同一台電腦/主機）**：技術難度低，但限制玩家聚集在一起
- **線上多人（遠端連接）**：技術難度高 2-3 倍，需要網路同步、延遲補償
- 建議：**先做本地多人版本**，驗證機制後再考慮線上版本

## 7. 針對本專案的方向建議

### 7.1 核心問題：你想要哪種「合作感」？

基於「原始人狩獵長毛象」的比喻，可以細分為兩種詮釋：

**詮釋 A：「每個人都有不可缺少的工具」**  
→ 選擇模式 A（非對稱角色）  
→ 像 Deep Rock Galactic 或 Monster Hunter  
→ 合作感來自「我的角色讓整個獵殺成為可能」

**詮釋 B：「整個流程需要每個人同時投入」**  
→ 選擇模式 B（任務流程）  
→ 像 Overcooked（但主題是狩獵）  
→ 合作感來自「少了任何一個人就做不完這整件事」

**建議：混合模式 A + B**  
- 玩家有不同角色（模式 A）
- 但「狩獵流程」需要所有角色同時執行不同任務（模式 B 的流程感）

### 7.2 美術方向建議

基於成本、可行性、市場差異化的三重考量：

**推薦：剪影風格（黑色剪影 + 少量顏色光效）**

原因：
1. 比像素風格節省 30-40% 製作時間
2. 視覺衝擊力強，市場差異化明顯（像《Limbo》但有色彩和活力）
3. 適合表現「原始/自然」主題
4. 可以逐步增加細節而不影響核心遊戲感
5. 4 個角色用剪影形狀區分比依靠顏色更直觀

### 7.3 技術可行性摘要

| 功能 | Godot 4 可行性 | 難度 |
|------|--------------|------|
| 本地 4 人控制 | ✅ 原生支援多輸入 | 低 |
| 非對稱角色系統 | ✅ 可行 | 中等 |
| 獵物 AI（行為樹）| ✅ 可行 | 中高 |
| 場景機關/陷阱系統 | ✅ 可行 | 中等 |
| 共享狀態/資源 | ✅ 可行 | 低 |
| 視差背景 | ✅ 原生支援 | 低 |
| 線上多人 | ⚠️ 可行但複雜 | 很高 |

## 結語：下一個設計問題

基於以上研究，我需要請教您一個關鍵問題，這個問題的答案將決定整個設計方向：

**您希望遊戲中的「合作」感來自哪裡？**

選項 A：「每個玩家有獨特技能，缺了誰都不行」  
→ 類似 Deep Rock Galactic，4 個差異化角色，自然形成合作  
→ 優點：每個玩家都有「英雄感」；缺點：需要設計 4 套平衡的技能組

選項 B：「任務本身就需要 4 個人同時做不同的事」  
→ 類似 Overcooked，強制分工，角色可以相似  
→ 優點：設計更直觀；缺點：容易感覺像在按照程序執行

選項 C：「混合—玩家有不同傾向，但執行任務時必須即時分工」  
→ 類似 Monster Hunter，每個人有武器特色，但在戰鬥中自然分工  
→ 優點：最接近「長毛象氛圍」；缺點：設計複雜度最高

> **研究目標**：廣泛調查 2015–2024 年間「玩家共同達成目標」的合作遊戲，分析各自的設計機制、視覺風格、可行性與製作成本

### 9.1 重新定義「合作感」的來源

用戶修正的核心觀點：
> 「長毛象」只是一個比喻，真正的重點是**玩家共同擊敗或達成某件事的行為本身**——這可以是任何形式的目標，不限於狩獵。

因此，本節從更廣的視角分析「集體達成目標」的合作遊戲，並從中提取可用於本專案的設計元素。

### 9.2 五款關鍵參考遊戲的深度分析

---

#### 🎮 遊戲 1：Lovers in a Dangerous Spacetime（2015，Asteroid Base）

![Lovers in a Dangerous Spacetime 截圖](img/lovers.png)

**Steam 評價**：非常正面（6,000+ 評論）  
**玩家數**：1-4 人  
**核心機制**：飛船內有 6+ 個戰鬥站（武器、護盾、引擎、超武），玩家數量永遠少於系統數量

**為什麼它讓合作感強烈：**
- **瓶頸公式**：`系統數量 (6+) > 玩家數量 (4)` → 永遠有事情沒人做
- **無固定角色**：任何人可以操作任何系統，威脅改變時大家立刻重新分工
- **共享物理空間**：所有人在同一個畫面，合作行為「看得見」
- **合作是溝通**：成功取決於協調，而非個人技術

**視覺風格分析：**
- 2D 俯視角，圓弧飛船設計（可以清楚看到所有房間）
- 鮮豔飽和色彩，粒子效果豐富
- 角色很小但用顏色區分

**Godot 4 技術難度：★★★☆☆ 中等**

| 系統 | 難度 | 說明 |
|------|------|------|
| 飛船房間佈局 | 低 | 靜態 2D 場景 |
| 玩家物理移動 | 低 | 標準 2D 移動 |
| 多站點狀態機 | 中 | 各站點有獨立邏輯 |
| 4 人同螢幕輸入 | 低 | Godot 原生支援 |
| 敵人 AI | 中高 | 需要方向感和波次設計 |

**製作成本估算（獨立開發）：**
```
美術：200-350h（飛船設計 + 敵人 + 特效）
開發：300-400h（Godot 4）
設計：80h
測試：50h
總計：約 630-880h（4-6 個月）
```

#### 🎮 遊戲 2：This Means Warp（2023，Outlier / Jagex）

![This Means Warp 截圖](img/this_means_warp.png)

**Steam 評價**：多半好評  
**玩家數**：1-4 人  
**核心機制**：Lovers 的精神繼承者，加入 **Roguelite 升級系統**——每局可以選擇升級不同船艦系統，改變本局的合作策略

**超越 Lovers 的地方：**
- **元策略層**：跑局前選擇升級路線，增加重玩性和預謀感
- **程序生成地圖**：不依賴手工關卡，每局都不同
- **敵人弱點系統**：不同敵人需要不同的應對策略，促進即時溝通

**視覺風格分析：**
- 較深色調，科幻風格更強
- 比 Lovers 更複雜的飛船佈局

**Godot 4 技術難度：★★★★☆ 中高**

| 新增複雜度 vs Lovers | 難度 |
|-------------------|------|
| Roguelite 升級系統 | 中高 |
| 程序生成關卡結構 | 高 |
| 敵人弱點多樣化 | 中 |

**製作成本估算（獨立開發）：**
```
比 Lovers 版本多 +150-200h（程序生成 + 升級系統）
總計：約 800-1100h（5-7 個月）
```

#### 🎮 遊戲 3：Ship of Fools（2022，Fika Productions）

![Ship of Fools 截圖](img/ship_of_fools.png)

**Steam 評價**：非常正面  
**玩家數**：2 人  
**核心機制**：兩人操作一艘船，分別控制左右砲台，管理甲板維修，在不斷湧來的海怪中生存

**設計特點：**
- **簡化版 Lovers**：專注在「戰鬥節奏」而非系統管理
- 視覺語言清晰，玩家永遠知道威脅在哪裡
- 每局有卡牌式升級選擇

**視覺風格分析：**
- 暖色調，卡通風格，高對比度
- 可愛的美術，即使混戰也視覺清晰

**Godot 4 技術難度：★★★☆☆ 中等**（比 Lovers 更簡單，系統少）

**製作成本估算：**
```
美術：150-250h（比 Lovers 少，系統更集中）
開發：250-350h
總計：約 500-700h（3-5 個月）
```

#### 🎮 遊戲 4：Lethal Company（2023，Zeekerss）⭐ 新增重要案例

![Lethal Company 截圖](img/lethal_company.png)

**Steam 評價**：史上最好評（300,000+ 評論，極度好評）  
**玩家數**：1-4 人  
**開發者**：1 人獨立開發  
**核心機制**：在廢棄星球上收集廢料，完成公司的配額目標，在怪物和環境威脅中生存並帶著戰利品回到船上

**為什麼這個案例非常重要（對本專案）：**

| 特點 | 說明 | 啟示 |
|------|------|------|
| **1人開發** | 整個遊戲由 1 個人做出來 | 獨立開發可行性的最佳證明 |
| **共同目標** | 配額制度——每個人都為同一個數字努力 | 集體目標比個人目標更凝聚 |
| **分工自然產生** | 沒有規定角色，船上監控者和探索者自然形成 | 合作感從情境中自然誕生 |
| **失敗即內容** | 死亡搞笑而非挫敗 | 降低失敗的心理成本 |
| **近距離語音** | Proximity voice chat 是靈魂 | 溝通機制可以是核心機制 |

**視覺風格分析：**
- 3D 低多邊形，Low-poly 風格
- 刻意的低解析度感，強化廉價工業氛圍
- 但環境設計細節精準，恐懼感來自光線和聲音

**Godot 4 技術難度：★★★★★ 高**（3D）

> ⚠️ 注意：Lethal Company 是 3D，若本專案採 2D，則參考的是「目標設計」和「失敗文化」，而非技術實作。

#### 🎮 遊戲 5：PlateUp!（2022，Alastair Janse van Rensburg）

![PlateUp! 截圖](img/plateup.png)

**Steam 評價**：好評如潮（1.5M+ 銷售，初始只有 1 人開發）  
**玩家數**：1-4 人  
**核心機制**：不只是煮菜——玩家共同**設計並建造廚房**，然後在營業時間合作服務客人。兩個階段都需要合作。

**為什麼這個案例值得注意：**
- **規劃階段 + 執行階段** 的雙重合作——準備期讓玩家建立共識
- 1 人開發、1.5M+ 銷售——成本效益的極端案例
- 合作感來自「共同建造的成果」而非只是「同時執行的動作」

**視覺風格分析：**
- 3D，低多邊形但色彩鮮豔
- 從俯視角看廚房，視野清晰

**Godot 4 技術難度：★★★★☆ 中高**（建造系統複雜）

### 9.3 橫向比較：「達成目標」的不同設計哲學

| | Lovers | This Means Warp | Ship of Fools | Lethal Company | PlateUp! |
|--|--------|-----------------|---------------|----------------|----------|
| **目標性質** | 生存/到達 | 生存/到達 | 生存 | 配額/收集 | 建造/服務 |

### 9.1「站點管理合作」子類型 DNA 定義

這類遊戲的核心公式：

```
可操控系統數量 > 玩家人數
       ↓

**與 Overcooked 模式的根本差異**：

### 10.3 惡魔城風格多人遊戲：設計可行性分析

#### 10.3.1 「惡魔城風格」核心 DNA

| 元素 | 惡魔城原版定義 | 本專案的對應挑戰 |
|------|--------------|----------------|
| **地圖結構** | 城堡型非線性探索（Metroid-vania 分支） | 多人需要同步位置，純探索圖可能造成分散問題 |
| **敵人設計** | 固定模式 + 位置依存（守門員型） | 多人時需重新平衡敵人血量/攻擊模式 |

**關鍵設計問題**：惡魔城的單人精心設計與多人合作的「人數混亂」天然衝突。成功的多人惡魔城式遊戲都需要針對性設計，而不是「惡魔城 + 多人」的直接加法。

### 10.5 Tile 複合角色系統：技術可行性與成本

這是本次最重要的創新點，也是本專案最獨特的設計決定。

#### 10.5.1 系統原理

```
MRMOTEXT 8×8 tile → 在 TileMapLayer 上拼接 → 組成角色外形
                                ↓
                    角色的「外觀」= N 個 tile 的集合
                    角色的「物理」= 獨立 CharacterBody2D
                    角色的「動畫」= TileMapLayer 的 tile 替換
```

**核心問題**：TileMapLayer 是靜態的場景圖層，角色是動態移動的實體。  
→ 解決方案：**每個敵人自帶一個獨立的小 TileMapLayer 節點**，跟著 CharacterBody2D 移動。

#### 10.5.2 可行性評分

| 維度 | 評分 | 說明 |
|------|------|------|
| **美術可行性** | ⭐⭐⭐⭐⭐ 極高 | 已在沙盒驗證，視覺效果良好 |
| **設計彈性** | ⭐⭐⭐⭐⭐ 極高 | 換 tile 即換外觀，快速迭代 |
| **程式複雜度** | ⭐⭐⭐☆☆ 中等 | TileMapLayer 動態控制需要學習曲線 |
| **效能風險** | ⭐⭐⭐☆☆ 中等 | 大量敵人各帶 TileMap 可能有 Draw Call 問題（需測試） |
| **動畫成本** | ⭐⭐⭐⭐☆ 低-中 | 「換 tile」代替「幀動畫」，美術成本極低 |
| **碰撞設計** | ⭐⭐⭐☆☆ 中等 | 外觀 TileMap ≠ 物理 CollisionShape，需分開維護 |

#### 10.5.3 美術成本節省計算

**傳統像素角色動畫成本（參考市場價）：**
- 1個角色 × 4方向 × 8動畫 × 8幀 = 256 幀
- 外包費用：$200-800/角色
- 10個敵人 = $2,000-8,000

**Tile 複合方案成本：**
- 1個角色外觀設計 = 15-30 分鐘（在 Godot 沙盒拼接）
- 動畫 = 靠 tile 替換，不需單獨繪製幀
- 10個敵人 ≈ 5-8小時工作量
- 費用：$0（自製）

> **結論：Tile 複合方案在美術成本上節省 90%+ 的時間與金錢，是本專案最重要的技術資產。**

#### 10.5.4 已識別的風險與對策

| 風險 | 程度 | 對策 |
|------|------|------|
| 大量敵人時 Draw Call 爆炸 | 🔴 高 | 使用 CanvasItem → `use_parent_material`；或在無需染色時合併為單一 Sprite |
| 角色看起來「拼湊感」太強 | 🟡 中 | VfxMix 特效（slash/impact/血液）可掩蓋拼接感，讓戰鬥感很強 |
| 動畫過渡不自然 | 🟡 中 | 用 `AnimationPlayer` 控制 tile 切換時機，加入 VfxMix 煙霧/閃光做過渡 |
| 碰撞體積與視覺不符 | 🟢 低 | 用簡化的矩形碰撞體，玩家感知不到 1-2 像素偏差 |

### 10.10 網路模式詳細分析（給非技術背景的說明）

> 這一節專門解釋「本地多人 vs 線上多人」的差異，讓你能做出有依據的決定。

#### 10.10.1 兩種多人模式的比較

```
本地多人（Local Co-op）          線上多人（Online Multiplayer）
─────────────────────────────    ──────────────────────────────
玩家在同一台電腦前               玩家分散在不同地方
使用多個手把或鍵盤分區            透過網路連線
                                 
技術難度：★★☆☆☆                技術難度：★★★★★
開發時間：不需要額外工程          額外開發時間：+2~4 個月
BUG 類型：主要是遊戲設計BUG      BUG 類型：網路延遲/封包遺失/不同步
```

**生活化比喻：**
- **本地多人** = 朋友來你家，坐在同一個沙發上玩，問題只有「遊戲好不好玩」
- **線上多人** = 朋友在台灣各地，問題是「他的螢幕跟你的螢幕要完全同步」

#### 10.10.2 線上同步的實際困難（以你的惡魔城遊戲為例）

橫向動作遊戲的線上同步比其他類型難 3-5 倍，原因：

```
問題一：每幀都在移動
  玩家A：在台灣北部，網路延遲 30ms
  玩家B：在台灣南部，網路延遲 60ms
  
  玩家A跳躍 → 自己看到「立刻起跳」
  玩家B看到玩家A → 「慢了 30ms 才起跳」
  
  → 如果不處理：玩家之間看到的畫面不同，攻擊命中不一致

問題二：攻擊判定
  玩家A攻擊了敵人（本地判定：命中）
  網路資料送到伺服器 → 確認 → 送到敵人端
  這段延遲期間，敵人可能已經移動走了
  
  → 需要「回滾機制」或「伺服器仲裁」，兩者都很複雜

問題三：敵人AI同步
  所有玩家必須看到「同一隻敵人做同一件事」
  → 敵人AI必須在某一方（通常是Host）執行，然後廣播給其他人
  → 如果Host掉線？需要「Host遷移」機制
```

#### 10.10.3 Godot 4 提供了什麼？

Godot 4 內建了一套叫 **Multiplayer API（ENet/WebRTC）** 的網路框架：

| 功能 | Godot 4 支援狀況 | 學習成本 |
|------|----------------|---------|
| 基礎連線（Host/Join） | ✅ 內建，比較容易 | 低 |
| 變數同步（@export var 廣播） | ✅ `MultiplayerSynchronizer` 節點 | 中 |
| 遠端呼叫函數（RPC） | ✅ `@rpc` 裝飾器 | 中 |
| 延遲補償/回滾 | ❌ 需自行實作 | 極高 |
| 匹配系統（自動找到陌生人組隊） | ❌ 需要伺服器服務（Steam/Nakama） | 高 |

**結論：Godot 4 讓「基礎連線」變容易，但「高品質動作遊戲同步」仍然是硬工程。**

#### 10.10.4 業界小團隊的實際做法

| 做法 | 代表遊戲 | 為什麼選這個 |
|------|---------|------------|
| **先本地多人，上架後再加線上** | Cuphead（1人後加2人線上），Broforce | 可以先驗證玩法，之後再做高難度工程 |
| **只做本地多人** | Guacamelee 1（後來2才加線上） | 專注核心體驗，避免網路開發負擔 |
| **直接做線上，但延遲容忍度高** | Terraria, Stardew Valley | 這類遊戲節奏慢，延遲不敏感 |
| **直接做線上，動作類** | Risk of Rain 2, Dead Cells（後加多人） | 需要大量工程，通常有專門的網路工程師 |

> **橫向動作 + 精確物理 + 線上多人** 的組合是最難的。你的目標遊戲正是這個組合。

#### 10.10.5 給本專案的建議方案

```
建議：兩階段策略

階段一（MVP，0-6個月）：
  ├── 本地多人（2-4人，同台電腦）
  ├── 完整實作所有玩法機制
  ├── 驗證「全滅→存檔點」是否好玩
  └── 幾乎零網路工程成本

階段二（後期，6個月後）：
  ├── 加入線上多人（LAN / Steam）
  ├── 使用 Godot 4 的 MultiplayerSynchronizer
  └── 只在玩法已驗證後才投入這塊成本
```

**風險說明：**
- 若「一開始就做線上多人」→ 可能花 3 個月在網路 Bug，遊戲本身還沒做好
- 若「先本地多人」→ 可能最後發現「沒有線上多人賣不動」，但至少遊戲已完整

#### 10.10.6 你需要決定的核心問題

> 「你的目標玩家是朋友圈（熟人一起玩）還是陌生人媒合（Steam找人組隊）？」

| 目標玩家 | 建議 |
|---------|------|
| **朋友 + 實體派對** | 本地多人就夠，甚至可以不做線上 |
| **Steam 上找陌生人** | 必須做線上多人，但可以在 MVP 後加 |
| **兩者都要** | 先本地，再加線上 |
